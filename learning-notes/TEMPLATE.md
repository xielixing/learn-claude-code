# 学习记录模板

## 使用方式

后续每次完成一个有学习价值的项目演进任务时，在 `learning-notes/` 目录下新增一篇学习记录。

这份文件是模板和写作规范，不要使用时间戳命名。真正的笔记文件需要使用时间和问题简介命名。

推荐文件名：

```text
YYYY-MM-DD-HHmm-修复或实现的问题简介.md
```

示例：

```text
2026-05-16-1810-add-agent-config-loader.md
2026-05-16-1935-fix-cli-command-parsing.md
```

## 写作目标

学习记录要帮助以后的人快速理解：

- 修改前的现象是什么
- 应该从哪些文件开始看
- 修改思路是什么
- 修改后如何确认功能真的生效
- 本次实现可以沉淀成什么经验

## 记录结构

每篇学习记录建议使用下面结构：

```markdown
# 主题

## 解决了什么问题

说明本次任务的目标，不只写“改了代码”，还要写清楚用户想让项目具备什么能力。

## 解决之前怎么样

描述修改前的行为或限制。尽量写清楚：

- 输入什么会触发问题
- 会看到什么输出、报错或不符合预期的结果
- 为什么这个行为不满足当前目标

## 修改了哪里

列出关键文件和职责，不需要贴完整代码。重点说明每个文件为什么需要改。

## 修改思路

用学习者能理解的语言解释核心方案：

- 数据或流程从哪里进入
- 关键判断在哪里发生
- 新逻辑如何接入旧逻辑
- 为什么这样改比较贴合当前项目结构

## 解决之后怎么样

描述修改后的行为。尽量对比修改前：

- 现在输入什么
- 会得到什么结果
- 新功能或修复点具体体现在哪里

## 验证方式

记录实际运行过的命令、测试或手动操作，以及对应结果。

## 本次经验

沉淀可以复用的经验，比如：

- 如何定位类似问题
- 对项目结构的新理解
- 下次实现相似功能时可以沿用的模式
- 哪些坑需要避免
```

## 协作约定

后续协助实现功能时，回答和执行过程尽量遵循：

1. 先说明目标和当前现象。
2. 指出大概需要查看或修改的文件位置。
3. 说明修改前输入什么、会出现什么行为。
4. 再进行代码修改。
5. 修改后运行对应命令或操作，说明现在可以实现什么功能。
6. 如果用户要求沉淀经验，就按本模板新增一篇带时间戳的学习记录。
